KEY DESIGN DECISIONS
================================================================================

1. COMPUTE IN TRINO (not Go)
   - Rationale: Pushes calculation near data, avoids transfer, scales horizontally

2. TEMPORAL FOR ORCHESTRATION (not cron)
   - Rationale: Durable, retryable, observable, handles failures gracefully

3. POSTGRES AS CONTROL PLANE (not filesystem)
   - Rationale: Single source of truth, transactional, queryable

4. ICEBERG AS ANALYTICS STORE (not Postgres)
   - Rationale: Time-travel, partitioning, incremental refresh support

5. NATURAL KEY IDEMPOTENCY
   - Rationale: (tenant, metric, period) enables safe re-runs and backfills

6. EVENT-DRIVEN REFRESH
   - Rationale: Cube partitions refresh immediately after job completion
