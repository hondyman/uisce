FIX #3: Backend SQL Query Correction
─────────────────────────────────────
File:     backend/internal/api/api.go
Lines:    6336-6395 (getRelatedObjects function)
Change:   Simplified SQL query to fix UUID type error

  Error Fixed:
  ┌─────────────────────────────────────────────────────────────────────────────┐
  │ ERROR: function char_length(uuid) does not exist (SQLSTATE 42883)          │
  └─────────────────────────────────────────────────────────────────────────────┘

  Root Cause:
  - Query tried to call char_length() on ce.edge_type_id which is UUID type
  - PostgreSQL doesn't have char_length for UUID types

  Solution:
  ┌─────────────────────────────────────────────────────────────────────────────┐
  │ Old Query (BROKEN):                                                         │
  │ JOIN catalog_edge_types cet ON (                                            │
  │   (char_length(ce.edge_type_id) = 36                                       │
  │    AND ce.edge_type_id ~ '^[0-9a-fA-F0-9-]{36}$'                          │
  │    AND cet.id = ce.edge_type_id::uuid)                                     │
  │    OR cet.edge_type_name = ce.edge_type_id                                 │
  │ )                                                                            │
  │                                                                             │
  │ New Query (FIXED):                                                          │
  │ LEFT JOIN catalog_edge_types cet ON cet.id = ce.edge_type_id               │
  │ SELECT ... COALESCE(cet.edge_type_name, ce.edge_type_id::text) as ...      │
  └─────────────────────────────────────────────────────────────────────────────┘

  Effect:  Backend now returns valid data without SQL errors
