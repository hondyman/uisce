# Keycloak / OIDC Setup — `uisce` Realm

## TL;DR — Fix the "GET .../realms/uisce/.../auth 400 (Bad Request)" error

The frontend (`oidc-client-ts`) is sending a PKCE-protected authorization
request to a Keycloak realm that doesn't exist yet, so Keycloak rejects it
with **HTTP 400**. Run this **once** after Keycloak first becomes reachable:

```bash
make keycloak-setup
# or, with non-default values:
KEYCLOAK_URL=https://100.84.50.65:8443 \
KEYCLOAK_ADMIN=admin \
KEYCLOAK_ADMIN_PASS=password \
./scripts/setup-keycloak-realm.sh
```

Then refresh the browser at `http://localhost:5173` and the OIDC login
should redirect you to the Keycloak login page (no more 400).

---

## Background

The browser console was showing:

```
GET https://100.84.50.65:8443/realms/uisce/protocol/openid-connect/auth
    ?client_id=semlayer-frontend
    &redirect_uri=http%3A%2F%2Flocalhost%3A5173%2Fauth%2Fcallback
    &response_type=code
    &scope=openid+profile+email
    &state=...
    &code_challenge=4ZgZEG5XWsej-I1cUt32trJ0nHQ6yBTAde_cQ_5TO5s
    &code_challenge_method=S256
→ 400 (Bad Request)
```

This is generated by `oidc-client-ts` inside the React app
(`frontend/src/config/oidc.ts`). The library automatically enables
**PKCE** (`code_challenge_method=S256`) for any `response_type=code`
request — that part is correct.

The 400 comes from the Keycloak server side, not the library. Keycloak
returns 400 when the authorization request cannot be satisfied. The most
common causes are:

| Cause                                                              | Fix                                      |
|--------------------------------------------------------------------|------------------------------------------|
| The realm does not exist                                           | Create the realm (this script)           |
| The client does not exist in the realm                             | Create the client (this script)          |
| Client exists but PKCE is `none` and `code_challenge` was sent     | Set `pkce.code.challenge.method = S256`  |
| Client's `redirectUris` does not include the request's `redirect_uri` | Add the redirect URI                  |
| Client's `webOrigins` does not include the SPA origin               | Add `+` or specific origins             |

In this repo, `docker-compose.remote.yml` only starts a fresh Keycloak
container — it never imports a realm or pre-creates a client. The
`uisce` realm and `semlayer-frontend` client therefore don't exist
until you run **`scripts/setup-keycloak-realm.sh`**.

---

## What the script does

1. Waits for Keycloak's master realm to be reachable.
2. Authenticates as the bootstrap admin (`admin` / `password`) using the
   `admin-cli` client to get an admin token.
3. Creates the `uisce` realm if missing (idempotent).
4. Creates the `semlayer-frontend` public client with:
   - **PKCE S256** (`attributes.pkce.code.challenge.method = S256`)
   - **redirectUris** = `http://localhost:5173/auth/callback`
   - **webOrigins** = `http://localhost:5173, +`
   - **post-logout redirect URI** = `http://localhost:5173/login`
   - **standard flow** enabled, **implicit flow** disabled.
5. Also creates a `uisce-backend` bearer-only client for the Go API
   (matches the `KEYCLOAK_JWKS_URL` in the backend).
6. Adds `openid`, `profile`, `email`, and `roles` to default client
   scopes so `frontend/.env.local`'s `scope=openid profile email` works.
7. Hits the OIDC discovery document
   (`/realms/uisce/.well-known/openid-configuration`) to confirm the
   realm is live.

The script is **idempotent** — re-running it on an already-configured
realm only updates the existing client with the canonical settings.

## Verifying the fix

After `make keycloak-setup`:

```bash
# 1. Realm discovery doc should be 200
curl -sk -o /dev/null -w '%{http_code}\n' \
  https://100.84.50.65:8443/realms/uisce/.well-known/openid-configuration
# → 200

# 2. The PKCE authorization request (the one that was failing) should
#    now return 200 (Keycloak login page) instead of 400.
curl -sk -o /dev/null -w '%{http_code} %{redirect_url}\n' \
  "https://100.84.50.65:8443/realms/uisce/protocol/openid-connect/auth?client_id=semlayer-frontend&response_type=code&scope=openid+profile+email&redirect_uri=http%3A%2F%2Flocalhost%3A5173%2Fauth%2Fcallback&code_challenge=abc&code_challenge_method=S256"
# → 200
```

Then refresh `http://localhost:5173/login` — the Keycloak login page
should appear, and successful auth should redirect back to
`http://localhost:5173/auth/callback?code=…&state=…`.

## Troubleshooting

| Symptom                                                                  | Cause / Fix                                                                                          |
|--------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------|
| `Failed to obtain admin token`                                           | Wrong `KEYCLOAK_ADMIN` / `KEYCLOAK_ADMIN_PASS` (defaults are `admin` / `password`).                  |
| `Realm 'uisce' already exists — skipping creation`                      | Good — script is idempotent.                                                                         |
| Login page renders, but submit returns **"Invalid redirect_uri"**        | Add the dev origin (`http://localhost:5173`) to the client's `redirectUris` via the admin console.  |
| Browser console: **"PKCE code verifier not found"** after callback       | `localStorage` was cleared between `/auth/callback` load and `handleCallback()`. Don't open the callback URL in a new tab. |
| **CORS** error in browser                                                | Add the SPA origin to the client's `webOrigins` (script does this with `+`).                        |

## Optional: import the static realm export

If you prefer declarative provisioning, the same realm can be imported
with `kc.sh`:

```bash
docker exec -i uisce-keycloak \
  /opt/keycloak/bin/kc.sh import --file -
  < keycloak/realm-uisce.json
```

Or via the admin REST API:

```bash
ADMIN_TOKEN=$(curl -sk -X POST \
  'https://100.84.50.65:8443/realms/master/protocol/openid-connect/token' \
  -d 'grant_type=password&client_id=admin-cli' \
  -d 'username=admin&password=password' | jq -r .access_token)

curl -sk -X POST \
  -H "Authorization: Bearer ${ADMIN_TOKEN}" \
  -H 'Content-Type: application/json' \
  --data @keycloak/realm-uisce.json \
  https://100.84.50.65:8443/admin/realms
