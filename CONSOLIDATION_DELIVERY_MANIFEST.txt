PROJECT SCOPE:
──────────────────────────────────────────────────────────────────────────────
Consolidate 12 metrics_registry tables and 8 dax_functions tables from domain
schemas in the alpha database (localhost:5432) into unified tables in the
public schema, reducing duplication and improving maintainability.

KEY METRICS:
──────────────────────────────────────────────────────────────────────────────
• Source Tables:        12 metrics_registry + 8 dax_functions = 20 total
• Total Records:        264 metrics + ~40-60 dax functions
• Domain Schemas:       17 (banking, capital_markets, currency_fx, etc.)
• Storage Reduction:    ~94% for these specific tables
• Migration Time:       <1 second
• Downtime Required:    0 minutes
• Data Loss Risk:       None (safe ON CONFLICT DO NOTHING)
• Implementation Time:  2-3 hours total
• Re-runnable:          Yes (fully idempotent)

CONSOLIDATION TARGETS:
──────────────────────────────────────────────────────────────────────────────

NEW TABLE: public.metrics_registry
  ├─ id (SERIAL PRIMARY KEY)
  ├─ node_id (VARCHAR 255)
  ├─ schema_domain (VARCHAR 100) ← NEW: tracks which domain
  ├─ category, description, formula_type, formula
  ├─ arguments (JSONB), badge, function_class, functions_used
  ├─ governance_status, audience, tags
  ├─ created_at, updated_at
  ├─ UNIQUE(node_id, schema_domain)
  └─ INDEXES: schema_domain, node_id, category

NEW TABLE: public.dax_functions
  ├─ id (SERIAL PRIMARY KEY)
  ├─ name (VARCHAR 100)
  ├─ schema_domain (VARCHAR 100) ← NEW: tracks which domain
  ├─ class (VARCHAR 50), badge (VARCHAR 10), description
  ├─ created_at
  ├─ UNIQUE(name, schema_domain)
  └─ INDEXES: schema_domain, name

IMPLEMENTATION PATHS:
──────────────────────────────────────────────────────────────────────────────

OPTION A: Quick Implementation (2-3 hours)
  Day 1:
    • Analyze + Backup + Migrate DB + Find Code + Update All Code + Test + Deploy
  Pros: Done quickly, single deployment
  Cons: Requires coordinating all code changes at once

OPTION B: Staged Implementation (1-2 days)
  Day 1: Analyze + Backup + Migrate DB + Create Backwards Compat Views
  Day 2: Update Code Incrementally + Test + Deploy
  Pros: Less risk, gradual rollout, backward compatible during transition
  Cons: Takes longer, views are temporary
